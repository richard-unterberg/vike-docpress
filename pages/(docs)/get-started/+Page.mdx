# Get Started

This page is a verbose MDX skeleton you can copy when writing Vike documentation.

It shows the common building blocks you usually want in a docs page: headings, paragraphs, inline code, links, lists, quotes, code blocks, tables, and expandable details. It intentionally stays self-contained and doesn't import any custom TSX components.

## What This Page Covers

- A short introduction.
- A minimal setup example.
- Internal and external links.
- Unordered and ordered lists.
- Quotes and notes.
- Code blocks for multiple file types.
- A table for comparing options.
- A collapsible section for secondary details.

## Minimal Example

The example below follows the style commonly used in Vike docs: keep the explanation short, show the file path, and use `satisfies Config` so the shape stays type-safe.

```ts
// pages/+config.ts

import type { Config } from 'vike/types'

export default {
  title: 'My Vike app',
  description: 'A minimal Vike setup'
} satisfies Config
```

You can then add a page:

```tsx
// pages/index/+Page.tsx

export default function Page() {
  return <h1>Hello</h1>
}
```

## Links

Use normal markdown links for docs prose.

- Internal link: [bodyAttributes](/bodyAttributes)
- External link: [Vike documentation](https://vike.dev)
- API reference: [Config settings](https://vike.dev/config)
- Source code: [Vike on GitHub](https://github.com/vikejs/vike)

If you mention a file, prefer making the file path explicit in the surrounding text. That makes the page easier to scan.

## Writing Style

Vike docs are usually direct and technical.

Write short paragraphs. Show the file path. Prefer one small example over one large example. Explain the default behavior first, and only then mention variations or caveats.

> Note
>
> If a feature only applies on the server, say so immediately. If it is cumulative, global, or overridden by another setting, say that near the top instead of burying it later.

## Recommended Structure

1. Start with one sentence that says what the feature does.
2. Show the smallest working example.
3. Explain when the setting runs or where it applies.
4. Mention edge cases and limitations.
5. End with related links.

## Example File Layout

The following example shows a small Vike app structure:

```text
pages/
├── +config.ts
├── index/
│   └── +Page.tsx
└── about/
    ├── +Page.tsx
    └── +route.ts
```

## Common Patterns

- Use `pages/+config.ts` for app-wide defaults.
- Use `pages/some-page/+config.ts` for page-specific overrides.
- Use `+route.ts` when you need routing that isn't captured by the directory name.
- Keep examples minimal unless the extra detail teaches something important.

## Caveats

- Prefer absolute doc links when linking outside your own site.
- Prefer relative doc links when linking to another page in the same docs app.
- Avoid mixing too many ideas in one example.
- Avoid hiding the main example below several screens of explanation.
